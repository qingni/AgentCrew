⏺ 分析结果：三者的角色分工                                                                                                                                                                                           
                                                                                                                                                                                                                     
  在这个项目中，Cursor CLI、Codex、Claude 是对等的"执行工具"，没有固定的角色分工——具体做什么完全由用户或 AI 编排器决定。但从代码中可以看出设计意图和典型用法：                                                       
                                                                                                                                                                                                                     
  三者的定位                                                                                                                                                                                                         
                                                                                                                                                                                                                     
  ┌────────┬──────────────────────────────────┬──────────────────────────┬──────────────────────┐                                                                                                                    
  │  工具  │             CLI 命令             │         默认模型         │       典型角色       │
  ├────────┼──────────────────────────────────┼──────────────────────────┼──────────────────────┤
  │ Codex  │ codex exec                       │ gpt-5-codex, o3, gpt-4.1 │ 默认工具，主要做编码 │
  ├────────┼──────────────────────────────────┼──────────────────────────┼──────────────────────┤
  │ Claude │ claude --print                   │ sonnet, opus, haiku      │ 编码 + 审查          │
  ├────────┼──────────────────────────────────┼──────────────────────────┼──────────────────────┤
  │ Cursor │ agent -p -f / cursor-agent -p -f │ auto, gpt-5, sonnet-4    │ 编码 + 审查          │
  └────────┴──────────────────────────────────┴──────────────────────────┴──────────────────────┘

  Cursor CLI 在项目中的具体角色

  1. 作为编排步骤中的可选执行工具：在任务编排（Orchestration）中，每个阶段的每个步骤都可以独立选择使用哪个工具。Cursor 是三选一之一。
  2. 典型场景是"复审"：从 UI 引导文案 (AutoOrchestrationPlannerSheet.swift:19) 可以看出设计者预期的典型流水线：
  "先 codex 编码，再 claude + cursor 复审，失败回滚后重试"

  2. 即 Codex 做主力编码，Cursor 和 Claude 做 code review / 复审。
  3. 由 AI 规划器自动分配：当用户用自然语言描述需求时，AI 规划器（AppViewModel.swift:1966）会自动拆解任务并为每个步骤推荐工具（recommendedTool）。工具解析逻辑在 AutoPlannerModels.swift:78-84：
    - 包含 "cursor" 或 "agent" → .cursor
    - 包含 "codex" 或 "openai" → .codex
    - 包含 "claude" → .claude
    - 默认 fallback → .codex

  调用方式的差异

  - Codex：codex exec --cd <dir> --sandbox workspace-write --skip-git-repo-check < prompt_file — 通过 stdin 传入 prompt
  - Claude：claude --print --permission-mode bypassPermissions --add-dir <dir> < prompt_file — 通过 stdin 传入 prompt
  - Cursor：agent -p -f "$(cat prompt_file)" — 通过命令行参数传入 prompt（用 cat 读文件内容作为参数）

  总结

  这三个工具在架构上是完全对等、可互换的执行引擎。项目的核心价值不在于某个特定 AI 工具，而是作为一个编排层，让用户可以在一条流水线中混合使用多个 AI CLI，实现"编码→审查→修复→重试"这样的多步自动化流程。Codex
  是默认的主力编码工具，Cursor 和 Claude 更多被用在复审、验证等辅助环节。

执行方式：基于 DAG 的波次并行                        
                                                                                                                                                                                                                     
  既不是纯串行，也不是无脑全并行，而是一个 依赖驱动的并行调度器（类似 CI/CD pipeline）。                                                                                                                             
                                                                                                                                                                                                                     
  核心逻辑（AppViewModel.swift:1376-1548）                                                                                                                                                                           
                  
  while 还有未完成的步骤:
      1. 找出所有依赖已满足的步骤 → 组成一个 "wave"
      2. 用 withTaskGroup 并行执行这一波所有步骤
      3. 等这一波全部完成后，更新完成集合
      4. 进入下一波

依赖关系来源

  依赖由两种方式产生（AppViewModel.swift:885）：

  1. 显式依赖：步骤的 dependsOnStepIDs（DAG 连线）
  2. 隐式依赖：如果阶段的 executionMode == .sequential，同阶段内每个步骤自动依赖前一个步骤

  实际执行效果举例

  假设有这样一个编排：

  阶段1 (parallel): [Codex编码A, Codex编码B]
  阶段2 (sequential): [Claude复审, Cursor复审]

  执行顺序：
  - Wave 1：Codex编码A + Codex编码B → 并行
  - Wave 2：Claude复审 → 串行（因为阶段2是sequential）
  - Wave 3：Cursor复审 → 串行（依赖Claude复审完成）

  失败处理

  - 步骤的 continueOnFailure = false（默认）→ 该步骤失败后整个 pipeline 停止（shouldStopPipeline = true）
  - continueOnFailure = true → 记录错误但继续执行后续波次

  总结：是一个 DAG 拓扑排序 + 波次并行 的调度模型，同一波内的步骤真正并行（Swift TaskGroup），波次之间串行等待。
